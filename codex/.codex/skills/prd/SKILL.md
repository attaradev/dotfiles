# Product Requirements Document

Use this skill to define a feature clearly before implementation starts.

## Workflow

1. Read the feature description and repository context.
2. Define goals, non-goals, user stories, scope, constraints, and success metrics.
3. Flag open questions instead of inventing requirements.
4. Follow `references/prd-template.md`.

## Quality rules

- Goals must be measurable.
- Non-goals must be explicit.
- User stories must cover happy path, errors, and edge cases.
- Open questions should be specific and owned.

## Resources

- `references/prd-template.md` provides the PRD structure and scope guidance.
