# Release Notes

Use this skill to turn recent work into clear, user-facing release notes.

## Workflow

1. Read the commit history, changelog, and any provided feature context.
2. Group work by theme or product area.
3. Write notes in user voice, not developer voice.
4. Include breaking changes and deprecations only when they matter to users.
5. Omit internal refactors, test-only work, and CI churn.

## Quality rules

- Answer "so what?" for every item.
- Include migration steps for breaking changes.
- Group related items under headings instead of flat lists.
- Keep the tone direct and release-ready.
